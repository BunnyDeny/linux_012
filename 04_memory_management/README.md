## 🌱 本章引言

前面两章一直在实模式下打转，用 `CS × 16 + IP` 这种简单粗暴的方式访问内存。但 x86 CPU 从 80386 开始，内存管理发生了翻天覆地的变化 🌊

i386 引入了两种全新的内存管理机制：

1. 🏗️ **段式管理**（Segmentation）—— 相比 8086 的"段寄存器 × 16"发生了重大变化
2. 📄 **页式管理**（Paging）—— 全新的机制，8086 完全没有

保护模式（Protected Mode）就是这两种机制**混合使用**的模式。在真正踏入保护模式之前，必须先理解这些基础概念，以及一个绕不过去的历史包袱 —— **A20 地址线问题** 🔌

> 🎯 **本章核心任务**：理解 i386 内存管理的基本框架，以及从实模式切换到保护模式前必须解决的 A20 地址线问题。

## 📖 概念速览

### 🏗️ 段式管理（Segmentation）

8086 的段式管理非常简单粗暴：

```
物理地址 = 段寄存器 × 16 + 偏移地址
```

i386 的保护模式下，段寄存器不再直接存放段基址，而是存放**段选择子**（Selector）。段的具体信息（基地址、限长、权限等）存放在内存中的**段描述符表**（GDT/LDT）里。

| 特性 | 8086 实模式 | i386 保护模式 |
|------|------------|--------------|
| 段寄存器内容 | 段基址 | 段选择子（Selector）|
| 段基址来源 | 段寄存器直接算 | 查段描述符表 |
| 段长度 | 固定 64KB | 可配置（1 字节 ~ 4GB）|
| 权限检查 | ❌ 没有 | ✅ 有（特权级、读写权限）|

### 📄 页式管理（Paging）

页式管理是 i386 新增的机制，8086 完全没有。

它将物理内存划分为固定大小的**页框**（Page Frame，通常 4KB），通过**页目录**和**页表**进行映射。程序看到的地址不再是物理地址，而是**虚拟地址**，由硬件动态翻译成物理地址。

> 💡 页式管理的最大好处：**隔离不同程序的内存空间**。即使两个程序使用了相同的虚拟地址，页表也能把它们映射到不同的物理页框，互不干扰。

### 🛡️ 保护模式 = 段式 + 页式

i386 的保护模式不是非此即彼的选择，而是**段式管理和页式管理的混合**：

1. 先用**段式管理**把虚拟地址转换成"线性地址"
2. 如果开启了分页，再用**页式管理**把线性地址转换成物理地址
3. 如果没开启分页，线性地址就是物理地址

```
逻辑地址（程序视角）
    ↓
段式管理（查 GDT/LDT）
    ↓
线性地址
    ↓
页式管理（查页目录/页表）← 可选，可开可关
    ↓
物理地址（最终写到地址总线上）
```

## 🔌 A20 地址线问题

### 📜 历史背景

1981 年，IBM 推出第一台 PC，使用的 CPU 是 **Intel 8088**。8088 只有 **20 根地址线**（A0 ~ A19），最大寻址空间是 1MB（`2^20 = 1,048,576` 字节）。

20 根地址线能表达的最大地址是 `0xFFFFF`（二进制 20 个 1）。但 8088 的段式寻址允许组合出 `0xFFFF:0xFFFF`：

```
0xFFFF × 16 + 0xFFFF = 0xFFFF0 + 0xFFFF = 0x10FFEF
```

这个地址有 **21 位**（`0x10FFEF`），超出了 20 根地址线的表达能力！

### 🔄 8088 的"地址回卷"行为

8088 只有 20 根地址线，第 21 位（A20）根本不存在。所以当地址超过 `0xFFFFF` 时，高位的 1 被默默丢弃：

```
0x10FFEF → 丢弃 A20 → 0x00FFEF
```

这种现象叫做**地址回卷**（Address Wrap-around）。程序如果"故意"利用这个特性，可以让 `0xFFFF:0xFFFF` 指向 `0x00FFEF`，而不是真的 1MB 之外。

> 💡 当时确实有一些程序利用了这个回卷特性来工作。比如某些内存检测程序、某些 DOS 下的技巧代码等。

### 🆕 80286 不再回卷了

1985 年，IBM 推出 AT 机，CPU 升级为 **Intel 80286**。80286 有 **24 根地址线**，最高可寻址 16MB。

关键是：**80286 不会在地址超过 1MB 时自动回卷了！**

```
0xFFFF:0xFFFF = 0x10FFEF → 80286 真的会去访问 0x10FFEF（第 21 位有效）
```

这本来是好事 —— 终于可以真正访问 1MB 以上的内存了。但问题出现了：**那些依赖地址回卷的老程序在 80286 上会崩溃！** 💥

### 🔧 IBM 的兼容方案：A20 开关

为了兼容那些依赖回卷特性的老程序，IBM 想了一个办法：**用一个开关来控制 A20 地址线是否有效**。

| 开关状态 | A20 地址线 | 行为 | 兼容老程序？|
|---------|-----------|------|-----------|
| **关闭**（默认）| A20 强制为 0 | 超过 1MB 的地址自动回卷 | ✅ 兼容 |
| **打开** | A20 正常生效 | 可以访问真正的 1MB 以上地址 | ❌ 不兼容 |

这个开关通过键盘控制器的某个端口来控制（是的，很奇葩，历史遗留）。

### ⚠️ 为什么操作系统必须打开 A20？

```
实模式下的地址总线：
  A0 ~ A19 有效，A20 被强制为 0
  → 最大只能访问 0xFFFFF（1MB - 1）

保护模式下的地址总线：
  A0 ~ A31 全部有效
  → 需要访问 4GB 地址空间
```

如果操作系统从实模式切换到保护模式时**没有打开 A20**：

- CPU 以为自己访问的是 `0x100000`（1MB 处）
- 但由于 A20 被强制为 0，实际访问的是 `0x000000`（0 地址处）
- 保护模式下的内存映射完全错乱，系统立刻崩溃 💥

> 🎯 **结论**：从实模式切换到保护模式之前，**必须打开 A20 地址线**，否则 32 位地址总线形同虚设，保护模式根本无法正常工作。

## 🧠 关键概念回顾

| 概念 | 一句话解释 |
|------|-----------|
| **8086 段式管理** | `段寄存器 × 16 + 偏移 = 物理地址`，简单粗暴 |
| **i386 段式管理** | 段寄存器存选择子，查描述符表得基址和限长 |
| **页式管理** | 虚拟地址 → 页表 → 物理地址，程序间内存隔离 |
| **保护模式** | 段式 + 页式混合，支持权限检查和内存隔离 |
| **8088 地址回卷** | 20 根地址线溢出后高位丢弃，`0x10FFEF` 变成 `0x00FFEF` |
| **A20 地址线** | 控制第 21 根地址线是否生效的开关 |
| **为什么要开 A20** | 保护模式需要完整的 32 位地址总线，不开就访问错乱 |
