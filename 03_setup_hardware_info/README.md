## 🌱 本章引言

上一章中，bootsect 成功将 setup 模块加载到了 `0x90200` 并跳转执行 🎉 但 setup 的使命远不止打印一行字符串 —— 在切换到保护模式之前，它必须尽可能多地**收集硬件信息** 🔍 供后续内核初始化时使用。

实模式下有一个巨大的便利：**BIOS 中断** 🖥️ 操作系统内核一旦进入保护模式，就再也无法直接调用 BIOS 中断了。所以 setup 必须趁还在实模式时，把所有可能用到的硬件参数"一网打尽" 🕸️ 存放到内存的特定位置。

> 🎯 **本章核心任务**：利用 BIOS 中断获取光标位置、扩展内存大小、显卡信息、硬盘参数等硬件信息，并将它们保存到 `0x90000` 起始的内存区域。

## 🗺️ 硬件信息内存布局

setup 把收集到的所有硬件参数，统一存放在 `INITSEG = 0x9000`（物理地址 `0x90000`）开始的内存区域 📦

| 偏移 | 大小 | 内容 | 来源 |
|------|------|------|------|
| `0x00` | 2 B | **🖱️ 光标位置**（DH=行, DL=列） | `int 0x10, AH=0x03` |
| `0x02` | 2 B | **💾 扩展内存大小**（单位：KB） | `int 0x15, AH=0x88` |
| `0x04` | 2 B | **🖥️ 显示模式**（BH=页, AL=模式） | `int 0x10, AH=0x0F` |
| `0x06` | 2 B | **📐 屏幕列数**（AH=列） | `int 0x10, AH=0x0F` |
| `0x08` | 2 B | **🎨 显卡信息 AX**（已废弃） | `int 0x10, AH=0x12` |
| `0x0A` | 2 B | **🎨 显卡信息 BX**（BH=状态, BL=显存） | `int 0x10, AH=0x12` |
| `0x0C` | 2 B | **🎨 显卡信息 CX**（CH=特性, CL=开关） | `int 0x10, AH=0x12` |
| `0x80` | 16 B | **💿 第一块硬盘参数表（hd0）** | BIOS 数据区 `0x104` |
| `0x90` | 16 B | **💿 第二块硬盘参数表（hd1）** | BIOS 数据区 `0x118` |

> 💡 为什么选 `0x90000`？因为 bootsect 最初被加载到这里，当 setup 开始执行时，bootsect 的代码已经不会再被使用了。这块内存正好可以"废物利用" ♻️

## 🧩 前置知识：段寄存器与隐含操作数

### 📐 实模式地址计算

实模式下，CPU 用 **段寄存器 × 16 + 偏移** 计算物理地址：

```
物理地址 = 段基地址 × 0x10 + 偏移地址
```

| 符号 | 值 | 物理地址 |
|------|-----|----------|
| `INITSEG` | `0x9000` | `0x9000 × 16 = 0x90000` |
| `SETUPSEG` | `0x9020` | `0x9020 × 16 = 0x90200` |

### 🧩 四个段寄存器各自管什么？

| 寄存器 | 全称 | 作用 |
|--------|------|------|
| `CS` | Code Segment | CPU **取指令**时用 `CS:IP` |
| `DS` | Data Segment | **读数据**时用 `DS:SI`、`DS:[offset]` 等 |
| `ES` | Extra Segment | **字符串操作**的目的段（如 `movsb` 的 `ES:DI`）|
| `SS` | Stack Segment | **栈操作**时用 `SS:SP` |

> ⚠️ **极其重要**：x86 的字符串操作指令（`movsb`、`movsw`、`stosb`、`lodsb`）和内存访问指令（`mov reg, (mem)`）都是**隐含操作数**指令 —— 它们不写死段地址，而是自动读取当时的段寄存器。如果段寄存器设错了，数据就会读写到完全错误的位置！

### 🎯 为什么 `movw %ax, %ds` 这一步不能省？

```asm
movw $INITSEG, %ax
movw %ax, %ds        ← 这一步做什么？
```

setup 开头已经做了 `DS = CS = 0x9020`（setup 自己的段）。但我们要把硬件参数保存到 `0x90000`，所以必须把 `DS` 改成 `0x9000`。

后续所有 `movw %reg, (offset)` 形式的指令，实际写的都是 **`DS:offset`**：

| 代码 | 不写 DS 时（DS=0x9020）| 写了 DS 后（DS=0x9000）|
|------|------------------------|------------------------|
| `movw %dx, (0)` | 写到 `0x90200` ❌ 覆盖 setup 自己！| 写到 `0x90000` ✅ 正确！|
| `movw %ax, (2)` | 写到 `0x90202` ❌ 覆盖 setup 自己！| 写到 `0x90002` ✅ 正确！|

> 💡 **一句话总结**：DS 是数据段的默认段寄存器，所有 `(offset)` 形式的内存访问都用它。不改成 `0x9000`，参数就写到 setup 自己脸上了 💥

## 📜 setup.S 逐行详解

### 🧭 整体作用

1. 🔄 初始化段寄存器（`CS = DS = ES = 0x9020`）
2. 📝 打印 `"setup is running"`
3. 🔍 利用 BIOS 中断收集硬件信息
4. 💾 将结果保存到 `0x90000` 开始的内存区域
5. ✅ 打印 `"hardware info saved"`
6. 🔒 死循环

---

### 📖 代码逐段解析

#### 1️⃣ 初始化段寄存器

```asm
_start_setup:
    movw %cs, %ax
    movw %ax, %ds
    movw %ax, %es
```

- bootsect 通过远跳转 `jmpl $SETUPSEG, $0` 将 `CS` 设为 `0x9020`
- setup 将 `DS`、`ES` 也统一为 `0x9020`，确保字符串打印在同一个段内 🎯

> 🔗 **与前文联动**：这里的 `DS = 0x9020` 用于后续 `lodsb` 读字符串。如果 DS 不是 `0x9020`，`lodsb` 就会从错误的位置读字符串。

#### 2️⃣ 打印 `"setup is running"`

```asm
    movw $setup_msg, %si
    movb $0x0E, %ah
print_loop:
    lodsb
    testb %al, %al
    jz print_done
    int $0x10
    jmp print_loop
```

**`lodsb` 的隐含操作数：**

`lodsb` 自动从 **`DS:SI`** 读取一个字节到 `AL`，然后 `SI++`。此时 `DS = 0x9020`（由 `_start_setup` 开头设置），所以实际读的是 `0x90200 + setup_msg_offset`，也就是 setup 自己段内的字符串位置。

> ⚠️ 如果 DS 还是 `0x9000`（后面会被修改），这里就会从 `0x90000` 附近读到垃圾数据！

### 🔧 核心：获取硬件信息

#### 🖱️ 获取光标位置

```asm
    movw $INITSEG, %ax
    movw %ax, %ds        ← ⭐ 关键：DS 改为 0x9000
    movb $0x03, %ah
    xor  %bh, %bh
    int  $0x10
    movw %dx, (0)        ← 写到 DS:0 = 0x90000
```

**`int 0x10, AH = 0x03` —— 读取光标位置**

| 输入 | 值 | 含义 |
|------|-----|------|
| `AH` | `0x03` | 功能号：读取光标位置 |
| `BH` | `0x00` | 显示页号（第 0 页）|

| 输出 | 寄存器 | 含义 |
|------|--------|------|
| `DH` | 行号 | 光标所在行（从 0 开始）|
| `DL` | 列号 | 光标所在列（从 0 开始）|

- 第 2 行 `movw %ax, %ds` **必须做**，否则 `movw %dx, (0)` 会写到 `0x90200`（setup 自己的代码位置）💥
- 结果存入 `DS:0 = 0x9000:0 = 0x90000`
- `DX` 是一个 16 位寄存器，`DH` 是高 8 位（行），`DL` 是低 8 位（列）

---

#### 💾 获取扩展内存大小

```asm
    movb $0x88, %ah
    int  $0x15
    movw %ax, (2)        ← 写到 DS:2 = 0x90002
```

**`int 0x15, AH = 0x88` —— 获取扩展内存大小**

| 输入 | 值 | 含义 |
|------|-----|------|
| `AH` | `0x88` | 功能号：获取 1MB 以上的扩展内存大小 |

| 输出 | 寄存器 | 含义 |
|------|--------|------|
| `AX` | 内存大小 | 单位：**KB** |

- 结果存入 `DS:2 = 0x90002`（此时 `DS = 0x9000`，由前面光标位置代码设置）
- 例如：如果机器有 64MB 内存，AX 返回值约为 `63 * 1024 = 64512`（KB）
- ⚠️ 注意：此中断返回的是 **1MB 以上** 的扩展内存，不包括最开始的 1MB 常规内存

---

#### 🖥️ 获取显示模式

```asm
    movb $0x0f, %ah
    int  $0x10
    movw %bx, (4)        ← 写到 DS:4 = 0x90004
    movw %ax, (6)        ← 写到 DS:6 = 0x90006
```

**`int 0x10, AH = 0x0F` —— 获取当前显示模式**

| 输入 | 值 | 含义 |
|------|-----|------|
| `AH` | `0x0F` | 功能号：获取显示模式 |

| 输出 | 寄存器 | 含义 |
|------|--------|------|
| `AL` | 显示模式号 | 如 `0x03` = 80×25 彩色文本模式 |
| `AH` | 屏幕列数 | 如 `80` = 80 列 |
| `BH` | 当前显示页 | 页号 |

- `BX` 先存入 `0x90004`：`BH` = 页号，`BL` = 未使用（通常为 0）
- `AX` 再存入 `0x90006`：`AH` = 列数，`AL` = 显示模式

> 🔗 **与前文联动**：这里的 `(4)` 和 `(6)` 都依赖 `DS = 0x9000`。如果前面获取光标位置时漏了 `movw %ax, %ds`，这两行就会写到 `0x90204` 和 `0x90206`，直接把 setup 代码覆盖掉。

---

#### 🎨 获取显卡信息

```asm
    movb $0x12, %ah
    movb $0x10, %bl
    int  $0x10
    movw %ax, (8)        ← 写到 DS:8 = 0x90008
    movw %bx, (10)       ← 写到 DS:10 = 0x9000A
    movw %cx, (12)       ← 写到 DS:12 = 0x9000C
```

**`int 0x10, AH = 0x12, BL = 0x10` —— 获取显卡相关信息**

| 输入 | 值 | 含义 |
|------|-----|------|
| `AH` | `0x12` | 功能号：获取显卡信息 |
| `BL` | `0x10` | 子功能号：返回显存大小和视频状态 |

| 输出 | 寄存器 | 含义 |
|------|--------|------|
| `AX` | （已废弃）| 早期版本使用，现无意义 |
| `BH` | 视频状态 | `0x00` = 彩色模式 🌈，`0x01` = 单色模式 ⬛ |
| `BL` | 显存大小 | `0x00`=64KB, `0x01`=128KB, `0x02`=192KB, `0x03`=256KB |
| `CH` | 特性连接器 | 特性位信息 |
| `CL` | 开关设置 | 显卡配置开关 |

- `AX` → `0x90008`，`BX` → `0x9000A`，`CX` → `0x9000C`
- 三行都依赖 `DS = 0x9000`

> 💡 内核初始化控制台时会读取这些信息，用于确定显示方式和可用显存 🖼️

#### 💿 复制硬盘参数表（hd0）

```asm
    movw $0x0000, %ax
    movw %ax, %ds        ← ⭐ DS 先清零，为 ldsw 做准备
    ldsw (4 * 0x41), %si ← ⭐ ldsw 会同时修改 DS 和 SI！
    movw $INITSEG, %ax
    movw %ax, %es        ← ES = 0x9000
    movw $0x0080, %di
    movw $0x10, %cx
    rep
    movsb                ← 从 DS:SI 复制到 ES:DI
```

这是本章**最难理解**的一段代码 🧠 我们逐行拆解：

**背景知识：BIOS 数据区中的硬盘参数**

BIOS 在引导阶段会将硬盘参数存放在中断向量表（IVT）区域：

| 位置 | 物理地址 | 内容 |
|------|----------|------|
| `4 * 0x41 = 0x104` | `0x0000:0x0104` | **hd0 参数表的 far pointer**（段:偏移）|
| `4 * 0x46 = 0x118` | `0x0000:0x0118` | **hd1 参数表的 far pointer**（段:偏移）|

> ⚠️ 注意：这两个位置**不是**中断向量本身，而是 BIOS 临时借用 IVT 区域存放的硬盘参数表指针。有些资料说"硬盘参数存在 0x41 号中断处"，这种说法不准确 ❌

**`ldsw` 指令详解：**

`ldsw`（Load Pointer Using DS, Word）是 x86 的 `lds` 指令的 16 位版本：

```asm
ldsw source, %si
```

作用：从 `source` 指定的内存地址读取一个 **far pointer**（4 字节），然后：
- **低 2 字节** → `SI` 寄存器（偏移地址）
- **高 2 字节** → `DS` 寄存器（段地址）

所以执行完 `ldsw (4 * 0x41), %si` 后：
- `DS:SI` 就**直接指向**了 hd0 参数表的位置 🎯
- **DS 已经被 ldsw 修改了**，不需要再手动设置

> ⚠️ **这就是为什么前面要先 `movw $0x0000, %ax; movw %ax, %ds`** —— 这一步**绝对不是多余的**！
>
> `ldsw` 的 CPU 执行时序分为两步：
> 1. **读取源操作数**（从内存地址 `0x104` 读 far pointer）← 这一步用的是**旧的 DS**
> 2. **写入 SI 和 DS**（把读到的 far pointer 拆开写入）
>
> 如果不清零 DS，上一步残留的 `DS = 0x9000` 会让 CPU 从 `0x90000 + 0x104 = 0x90104` 读取，而不是正确的 `0x00104`！💥
>
> | 是否清零 DS | `ldsw` 读取的物理地址 | 结果 |
> |------------|----------------------|------|
> | ❌ 不清零（DS=0x9000） | `0x90000 + 0x104 = 0x90104` | 读到垃圾数据 |
> | ✅ 清零（DS=0x0000） | `0x00000 + 0x104 = 0x00104` | 读到正确的 far pointer |

**`rep movsb` 详解：**

| 指令 | 作用 |
|------|------|
| `movsb` | Move String Byte，从 **`DS:SI`** 搬运 **1 字节** 到 **`ES:DI`**，然后 `SI++`、`DI++` |
| `rep` | Repeat，重复执行直到 `CX = 0`，每执行一次 `CX--` |

- `CX = 0x10 = 16`，所以复制 **16 字节**
- **源地址**：`DS:SI`（由 `ldsw` 自动设置，指向 BIOS 中的 hd0 参数表）
- **目的地址**：`ES:DI = 0x9000:0x0080 = 0x90080`

> 🔗 **与前文联动**：
> - `DS` 由 `ldsw` 自动设置为 hd0 参数表所在段（不需要手动设）
> - `ES` 由代码显式设置为 `0x9000`（第 5~6 行）
> - `DI` 由代码显式设置为 `0x0080`（第 7 行）
> - `rep movsb` 是**隐含操作数**指令，自动读取当时的 `DS` 和 `ES`。如果 `ES` 没设对，数据就会搬错地方！

#### 💿 复制硬盘参数表（hd1）

```asm
    movw $0x0000, %ax
    movw %ax, %ds        ← ⭐ 再次清零 DS，因为上一步的 ldsw 修改了它！
    ldsw (4 * 0x46), %si ← ⭐ 再次修改 DS 和 SI
    movw $INITSEG, %ax
    movw %ax, %es
    movw $0x0090, %di
    movw $0x10, %cx
    rep
    movsb
```

- 和 hd0 的代码几乎完全一样
- 差别只有两处：源地址是 `4 * 0x46 = 0x118`，目的地址是 `0x0090`（`0x90090`）
- **注意：必须先 `movw $0x0000, %ax; movw %ax, %ds`** 把 DS 清零，因为上一步的 `ldsw` 已经修改了 DS

> 🔗 **与前文联动**：上一步 `ldsw` 已经把 DS 改成了 hd0 参数表所在的段（比如 `0xF000`）。如果不先清零 DS，第二次 `ldsw` 读取 `(4 * 0x46)` 时，CPU 会用 `DS = 0xF000` 去访问内存：
> 
> | 是否清零 DS | `ldsw` 读取的物理地址 | 结果 |
> |------------|----------------------|------|
> | ❌ 不清零（DS=hd0参数表所在段） | `0xF0000 + 0x118 = 0xF0118` | 读到 hd0 参数表附近的垃圾数据 💥 |
> | ✅ 清零（DS=0x0000） | `0x00000 + 0x118 = 0x00118` | 读到正确的 hd1 far pointer |

#### 🔍 检测第二块硬盘是否存在

```asm
    movw $0x1500, %ax
    movb $0x81, %dl
    int  $0x13
    jc   no_disk1
    cmpb $3, %ah
    je   is_disk1
```

**`int 0x13, AH = 0x15` —— 获取驱动器类型**

| 输入 | 值 | 含义 |
|------|-----|------|
| `AH` | `0x15` | 功能号：获取驱动器类型 |
| `DL` | `0x81` | 驱动器号：`0x80`=第一块硬盘，`0x81`=第二块硬盘 |

| 输出 | 条件 | 含义 |
|------|------|------|
| `CF = 1` | 进位标志置位 | 驱动器不存在或出错 ❌ |
| `AH = 0` | 成功返回值 | 没有目标驱动器 |
| `AH = 1` | 成功返回值 | 软盘（360KB）💾 |
| `AH = 2` | 成功返回值 | 软盘（1.2MB）💾 |
| `AH = 3` | 成功返回值 | **硬盘** 💿 |

```asm
    jc   no_disk1      ; CF=1 → 出错/不存在，跳转到 no_disk1
    cmpb $3, %ah       ; 比较 AH 和 3
    je   is_disk1      ; AH=3 → 是硬盘，跳转到 is_disk1（什么都不做）
```

**如果没有第二块硬盘（`no_disk1`）：**

```asm
no_disk1:
    movw $INITSEG, %ax
    movw %ax, %es        ← ES = 0x9000
    movw $0x0090, %di
    movw $0x10, %cx
    movw $0x00, %ax
    rep
    stosb                ← 将 AL=0 写入 ES:DI，重复 16 次
```

- `stosb`：Store String Byte，将 `AL` 中的值写入 **`ES:DI`**，然后 `DI++`
- `rep stosb`：重复执行 CX 次
- 作用：将 `0x90090` 开始的 16 字节全部清零 🧹
- 这样内核后续读取 hd1 参数时，看到全零就知道不存在第二块硬盘

> 🔗 **与前文联动**：`stosb` 使用 `ES:DI` 作为目的地址。这里的 `ES = 0x9000`（由第 2~3 行设置），`DI = 0x0090`，所以实际写入的是 `0x90090`。如果 `ES` 没设对，就会往错误的位置写零。

```asm
is_disk1:
    /* 什么都不做，继续执行后续代码 */
```

#### 4️⃣ 🔄 段寄存器状态变化时序

setup 执行过程中，DS 和 ES 的值在不断变化。理解这一点至关重要 🧠

| 阶段 | DS 的值 | ES 的值 | 用途 |
|------|---------|---------|------|
| 初始化 | `0x9020` | `0x9020` | 📝 打印字符串、执行 setup 代码 |
| 保存光标/内存/显卡信息 | `0x9000` | `0x9020` | 💾 `movw reg, (offset)` 写到 `0x90000` 区域 |
| 复制 hd0 参数 | **由 `ldsw` 设置** | `0x9000` | `DS:SI` 指向 BIOS 参数表，`ES:DI` 指向 `0x90080` |
| 复制 hd1 参数 | **由 `ldsw` 设置** | `0x9000` | `DS:SI` 指向 BIOS 参数表，`ES:DI` 指向 `0x90090` |
| 打印结束信息 | `0x9020` | `0x9000` | 📝 `lodsb` 读取 setup 段内的字符串 |

> ⚠️ **常见错误**：如果在 `is_disk1` 之后忘记把 `DS` 改回 `0x9020`，`lodsb` 读字符串时就会从 `0x90000` 附近读取垃圾数据 💥

```

    0x07C00 ─┬─ bootsect（512B）← BIOS 自动加载
             │

    0x07C00 ─┼─ （原位置，将被覆盖）

             │

    0x90000 ─┼─ 🖱️ 光标位置（2B）
    0x90002 ─┼─ 💾 扩展内存大小（2B）
    0x90004 ─┼─ 🖥️ 显示模式（4B）
    0x90008 ─┼─ 🎨 显卡信息（6B）
    0x90080 ─┼─ 💿 hd0 参数表（16B）
    0x90090 ─┼─ 💿 hd1 参数表（16B，若不存在则为 0）
             │
阶段四：setup 死循环，等待被覆盖或继续执行
    0x90200 ─┴─ setup 代码继续执行（后续章节将切换到保护模式）
```

## 🧠 关键概念回顾

| 概念 | 一句话解释 |
|------|-----------|
| **为什么要在实模式收集硬件信息？** | 进入保护模式后无法直接调用 BIOS 中断 🚫 |
| **`ldsw` 做了什么？** | 从内存读取 far pointer，低 2 字节给 SI，高 2 字节给 DS |
| **`rep movsb` 做了什么？** | 逐字节搬运 CX 次，自动更新 SI、DI；源=`DS:SI`，目的=`ES:DI` 🔄 |
| **`rep stosb` 做了什么？** | 将 AL 写入 `ES:DI`，重复 CX 次，自动更新 DI 🔄 |
| **`lodsb` 做了什么？** | 从 `DS:SI` 读 1 字节到 AL，然后 SI++ 🔄 |
| **为什么 `movw %ax, %ds` 不能省？** | `movw reg, (offset)` 默认用 DS 算地址；不改就写到 setup 自己脸上 💥 |
| **`int 0x13, AH=0x15` 返回值** | `AH=3` 表示硬盘存在，`CF=1` 表示出错/不存在 |
| **硬盘参数表存在哪里？** | BIOS 数据区 `0x0000:0x0104`（hd0）和 `0x0000:0x0118`（hd1）|
| **`0x90000` 为什么可以复用？** | bootsect 已经执行完毕，这块内存不会再被访问 ♻️ |
