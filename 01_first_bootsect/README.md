## 🔨 编译 & 运行

### 1️⃣ 编译

```bash
as --32 bootsect.S -o bootsect.o
ld -m elf_i386 -Ttext 0x0 --oformat binary bootsect.o -o bootsect
```

| 参数 | 作用 |
|------|------|
| `as --32` | 32 位模式汇编（16 位代码需要指定 32 位汇编器模式）|
| `-Ttext 0x0` | 代码段起始地址设为 `0x0000` |
| `--oformat binary` | 输出纯二进制文件，不要 ELF 格式 |

> 💡 最终生成的 `bootsect` 是纯二进制镜像，可以被 BIOS 直接识别执行。

### 2️⃣ 运行

```bash
qemu-system-i386 -fda bootsect
```

🎉 你会看到屏幕左上角用 **红色** 🔴 显示 `Hello World!`

### 3️⃣ 验证

```bash
ls -l bootsect
xxd bootsect | tail -3
```

- 文件大小恰好 **512 字节** 📏
- 最后两个字节是 `55 aa`（引导签名）✅

## 📜 bootsect.S 代码详解

这是一段 **x86 实模式引导扇区** 汇编代码，会被 BIOS 加载到内存 `0x7C00` 处执行。

### 🧭 整体作用

在屏幕上用红色显示 `"Hello World!"`，然后死循环 🔒

### 🧩 前置知识：CS 与 IP

在 x86 实模式下，CPU 通过 **CS:IP** 找到下一条要执行的指令。

| 寄存器 | 全称 | 作用 |
|--------|------|------|
| **CS** | Code Segment（代码段） | 决定"哪一段"内存 |
| **IP** | Instruction Pointer（指令指针） | 决定段内的"偏移位置" |

物理地址计算公式：

```
物理地址 = CS × 16 + IP
```

🌰 **例子**：`CS = 0x07C0, IP = 0x0000`

```
物理地址 = 0x07C0 × 16 + 0x0000 = 0x07C00
```

BIOS 把引导扇区加载到 `0x7C00`，但不同 BIOS 跳转前的 `CS:IP` 组合可能不同：

| BIOS 类型 | CS | IP | 物理地址 |
|-----------|-----|-----|----------|
| 类型 A 🔵 | `0x0000` | `0x7C00` | `0x7C00` |
| 类型 B 🟢 | `0x07C0` | `0x0000` | `0x7C00` |

虽然物理地址相同，但如果后续代码访问数据（如 `msg` 字符串）时段寄存器不一致，就会找错地方 💥

> 💡 因此引导代码的第一条指令通常是一条**远跳转**，强制统一 CS 为 `0x07C0`，确保所有段寄存器环境一致。

### 📖 逐段解析

#### 1️⃣ 常量与声明

```asm
BOOTSEG = 0x7c0
```
- `0x7C0` 是段地址。BIOS 把引导扇区加载到物理地址 `0x7C00`。
- 在 8086 的段:偏移寻址中，`0x7C0:0x0000` = `0x7C0 × 16 + 0` = `0x7C00` ✅

```asm
.code16
```
- 告诉汇编器生成 **16 位实模式** 代码。BIOS 启动时 CPU 处于实模式。

```asm
.text
.global _start
```
- `.text` 表示代码段。
- `.global _start` 将 `_start` 导出为全局符号。

#### 2️⃣ 远跳转（Far Jump）—— 初始化 CS:IP 🔑

```asm
_start:
    jmpl $BOOTSEG, $start2
```

这是 **关键的一行**！`jmpl` 是**远跳转**，同时修改 `CS` 和 `IP`：

- BIOS 加载引导扇区后，`CS:IP` 可能是 `0x0000:0x7C00` 或 `0x07C0:0x0000`，取决于 BIOS 实现。
- 这段远跳转确保 `CS = 0x7C0`，`IP = start2` 的偏移地址。
- 这样无论 BIOS 从哪种状态启动，代码的段地址都统一为 `0x7C0`，后续数据访问不会错乱 🎯

#### 3️⃣ 初始化数据段寄存器

```asm
start2:
    movw $BOOTSEG, %ax
    movw %ax, %ds
    movw %ax, %es
    movw %ax, %fs
    movw %ax, %gs
```

把所有数据段寄存器（`DS`, `ES`, `FS`, `GS`）都设为 `0x7C0`。

> 💡 为什么？因为后面要访问 `msg` 字符串，如果段寄存器不对，就找错地方了！

#### 4️⃣ 调用 BIOS 10h 中断显示字符串 🖥️

```asm
    movw $msg, %ax
    movw %ax, %bp        # BP = 字符串地址（偏移量）
    movw $0x01301, %ax   # AH = 0x13, AL = 0x01
    movw $0x0c, %bx      # BH = 0, BL = 0x0C（亮红色 🔴）
    movw $12, %cx        # CX = 字符串长度（"Hello World!" 共 12 个字符）
    movb $0, %dl         # DL = 0（起始列号）
    int $0x010           # 调用 BIOS 10h 视频中断
```

这是 **BIOS INT 10h, AH=13h** 功能：在指定位置显示字符串。

| 寄存器 | 值 | 含义 |
|--------|-----|------|
| `AH` | `0x13` | 显示字符串功能号 |
| `AL` | `0x01` | 写模式（更新光标）|
| `BH` | `0x00` | 显示页号 |
| `BL` | `0x0C` | 字符颜色属性（亮红色）|
| `CX` | `12` | 字符串长度 |
| `DH` | 未设置(0) | 行号 |
| `DL` | `0` | 列号 |
| `ES:BP` | 指向 `msg` | 字符串地址 |

> ⚠️ `DH` 没显式设置，默认为 0，所以字符串显示在屏幕左上角。

#### 5️⃣ 死循环

```asm
loop:
    jmp loop
```

无限跳转自身，CPU 就在这里卡住不动。引导扇区代码通常不会返回（没有地方可返回）。

#### 6️⃣ 数据区

```asm
msg:
    .ascii "Hello World!"
```

定义字符串数据，共 12 个 ASCII 字符。

```asm
.org 510
boot_flag:
    .word 0xaa55
```

- `.org 510`：强制将接下来的内容放在偏移 510 字节处。
- `0xAA55`：**引导扇区签名（Magic Number）** ✨
- BIOS 检查软盘/硬盘的第 0 扇区，如果最后两个字节是 `0x55 0xAA`（小端序存储），就认为这是**可引导扇区**。

整个扇区恰好 **512 字节**！📏

### 🤔 为什么代码会在第 0 扇区？

**不是运行时命令把代码"放到"那里的，而是 BIOS 开机后自动去第 0 扇区找！**

#### 🧱 什么是"第 0 扇区"？

软盘/硬盘的最小读写单位是**扇区**，每个扇区固定 **512 字节**。

```
虚拟软盘结构：
┌─────────┬─────────┬─────────┐
│  扇区0  │  扇区1  │  ...   │
│ 512字节 │ 512字节 │        │
│ LBA 0   │ LBA 1   │        │
└─────────┴─────────┴─────────┘
```

#### 📜 BIOS 的历史约定

IBM 在 1981 年定下的规矩：

> **开机后，BIOS 自动从启动设备的第 0 扇区读取 512 字节到内存 0x7C00，然后跳过去执行。**

```
开机流程：
BIOS 上电自检
    │
    ▼ ① 找启动设备（软盘/硬盘）
读取 512B 到 0x7C00
    │
    ▼ ② 检查最后两字节是否为 0x55 0xAA
   是 ✅ → 跳转到 0x7C00 执行
   否 ❌ → 报 "No bootable device"
```

#### 🖥️ QEMU 做了什么？

```bash
qemu-system-i386 -fda bootsect
```

`-fda` 把 `bootsect` 文件当作虚拟软盘 A 插入虚拟机。因为 `bootsect` 恰好 **512 字节**，它正好占满虚拟软盘的**第 0 扇区**。

```
你的命令              QEMU 内部
┌──────────┐        ┌─────────────────┐
│ bootsect │ ─fda─► │  虚拟软盘驱动器   │
│ (512字节)│        │  扇区0=bootsect │
└──────────┘        └─────────────────┘
                            │
                            ▼
                      BIOS 读取扇区0
```

> 💡 如果文件不是 512 字节，BIOS 仍然只读前 512 字节，超出的不会被自动加载。

| 问题 | 答案 |
|------|------|
| 为什么在第 0 扇区？ | **BIOS 开机后自动去第 0 扇区找** |
| 谁决定的？ | **IBM/BIOS 工程师在 40 年前决定的** 🕰️ |
| QEMU 做了什么？ | **把 `bootsect` 映射成虚拟软盘的第 0 扇区** |

## 🎯 启动流程图

```
BIOS 启动
   ↓
加载第 0 扇区到 0x7C00
   ↓
检查 0x7C00+510 处是否为 0x55AA ✅
   ↓
跳转到 0x0000:0x7C00（或 0x7C0:0x0000）
   ↓
远跳转到 0x7C0:start2（统一段寄存器）
   ↓
初始化 DS, ES, FS, GS
   ↓
调用 INT 10h 显示红色 "Hello World!"
   ↓
死循环 🔒
```
